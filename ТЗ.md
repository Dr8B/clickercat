# Техническое задание: ClickerCat

Кроссплатформенный автокликер с глобальной горячей клавишей и отправкой набора
нажатий клавиш клавиатуры и кнопок мыши за одну итерацию.

## 1. Назначение

Простое десктопное приложение, которое по нажатию заданной пользователем горячей клавиши
эмулирует нажатие набора клавиш клавиатуры и/или кнопок мыши. За одну итерацию нажимаются
все клавиши из набора, затем выдерживается задержка, затем — новая итерация.

## 2. Технологический стек

| Параметр | Значение |
|----------|----------|
| Язык | Java 25 (`--release 25`) |
| UI | Swing (javax.swing) |
| Сборка | Maven |
| ОС | Linux, Windows |
| Артефакт | Исполняемый «fat-jar» (запуск `java -jar clickercat.jar`) |

### 2.1. Зависимости

- **JNativeHook** (`com.github.kwhat:jnativehook`) — глобальный перехват клавиатуры/мыши.
  Критически важно: стандартный Swing `KeyListener` ловит события только когда окно
  в фокусе. Для глобальной горячей клавиши (срабатывает в фоне, поверх любой игры/программы)
  нужен системный хук — это JNativeHook.
- **java.awt.Robot** (входит в JDK) — эмуляция нажатий клавиш и кнопок мыши на уровне ОС.
- **SnakeYAML** (`org.yaml:snakeyaml`) — чтение/запись конфигурации в YAML.

## 3. Функциональные требования

### 3.1. Запуск
- Приложение запускается одним окном Swing.
- При старте загружается конфигурация из YAML (см. §5). Если файла нет — используются
  значения по умолчанию (§5.1) и файл записывается.

### 3.2. Горячая клавиша срабатывания (триггер)
- Пользователь задаёт **горячую клавишу** (триггер) через UI: кнопка «✏️», после
  нажатия которой приложение перехватывает следующую нажатую пользователем клавишу и
  сохраняет её как триггер.
- Это **одна** клавиша, **без модификаторов** (Ctrl/Shift/Alt не учитываются).
- Триггером может быть клавиша клавиатуры или кнопка мыши.
- Горячая клавиша работает **глобально** — даже когда окно приложения не в фокусе.

### 3.3. Режим работы (переключатель)
Переключатель режима:
- **Удержание (HOLD)** — кликер работает, пока горячая клавиша **зажата**; как только
  клавишу отпустили — кликер останавливается. Пока клавиша удерживается, итерации
  повторяются (автоповтор с задержкой `delayMs`).
- **Переключение (TOGGLE)** — нажали горячую клавишу — кликер запустился (работает);
  нажали ещё раз — остановился.

### 3.4. Набор клавиш для нажатия
- Пользователь формирует **список** клавиш и кнопок, которые будут нажиматься:
  произвольная комбинация клавиш клавиатуры и кнопок мыши.
- В UI: добавление в список (кнопка **«+»** + захват нажатия), удаление выбранного
  элемента (кнопка **«−»** либо клавиша `Delete` на выделенном элементе списка),
  отображение списка.
- Список **не может быть пустым** (минимум один элемент; значение по умолчанию — §5.1).

### 3.5. Эмуляция мыши
- Курсор мыши **не перемещается**. Клики выполняются в **текущей позиции курсора**
  (где он есть в момент клика). Координаты X/Y не задаются и не хранятся.

### 3.6. Задержка между итерациями
- Поле ввода задержки между итерациями в миллисекундах.
- Целое число в диапазоне **0–10000 мс**. Значение по умолчанию — 100 мс.
- Поле **не может быть пустым**; ввод вне диапазона/нечисловой — не принимается
  (откат к последнему валидному значению или к значению по умолчанию).

### 3.7. Аварийная остановка
- Отдельная **настраиваемая** горячая клавиша (по умолчанию — `ESC`), которая
  немедленно и принудительно останавливает кликер из любого состояния.
- Работает глобально (как и триггер).

### 3.8. Взаимоисключение клавиш (защита от конфликтов)
- Одну и ту же клавишу/кнопку **нельзя** использовать одновременно в нескольких ролях.
  Все роли должны быть различны:
  - триггер срабатывания (§3.2);
  - клавиша аварийной остановки (§3.7);
  - любой элемент списка нажатий (§3.4).
- При попытке назначить уже занятую клавишу в другую роль — операция отклоняется
  с понятным сообщением (это защищает от самоперехвата эмулированных событий и от
  рекурсии в режиме HOLD).
- Проверка выполняется **в пределах одного профиля** (§3.11): профили независимы,
  поэтому одна и та же клавиша может быть триггером в одном профиле и клавишей
  нажатия в другом — конфликта нет, ведь активен всегда только один профиль.

### 3.9. Сохранение настроек
- Все настройки сохраняются в YAML-файл `./clickercat.cfg.yaml` в **рабочей директории**
  приложения (рабочая директория = `new File("./")`, т.е. CWD процесса; **не** обязательно
  каталог jar-файла).
- Сохранение выполняется **при каждом изменении** любой настройки.
- **Перед каждым сохранением** текущий файл конфигурации копируется в бэкап
  `./clickercat.cfg.yaml.bak` (всегда один и тот же файл, перезаписывается).
- Загрузка: при старте.

### 3.10. Главный выключатель (Вкл/Выкл)
- Отдельная кнопка **«Вкл/Выкл»** — главный выключатель всего механизма
  (заменяет прежнюю кнопку «Старт/Стоп»).
- Когда механизм **выключен**:
  - текущее кликанье немедленно останавливается (нажатые клавиши отпускаются);
  - горячие клавиши (триггер §3.2 и аварийная остановка §3.7) **не срабатывают** —
    приложение «на паузе» и не мешает работе, в том числе когда свёрнуто в трей;
  - захват клавиш для назначения (триггер / аварийная остановка / список нажатий)
    **продолжает работать**.
- Когда механизм **включён** — всё работает штатно (§3.2, §3.3).
- Состояние сохраняется в конфиге (`enabled`, §5). Старые конфиги без этого поля
  трактуются как **включённые**.
- Статус приложения отражает три состояния:
  - **Отключён** — механизм выключен;
  - **Ожидание** — механизм включён, но кликер сейчас не работает;
  - **Активен** — кликер работает (идут итерации).

### 3.11. Профили

- Настройки сгруппированы в **именованные профили**: пользователь заводит несколько
  наборов настроек (для разных игр/задач) и переключается между ними.
- В профиль входит **всё, кроме главного выключателя** (§3.10): триггер (§3.2),
  аварийная остановка (§3.7), режим (§3.3), задержка (§3.6), список нажимаемых
  клавиш (§3.4) и флажок «Поверх всех окон».
- Глобальны (общие для всего приложения, вне профилей) только: `enabled` (§3.10)
  и указание на то, какой профиль активен.
- Активен всегда ровно один профиль; его настройки и определяют работу кликера.
- Управление профилями в UI: выпадающий список профилей и кнопки **«+»** (создать),
  **«−»** (удалить текущий) и **«✏️»** (переименовать).
  - Имя профиля — непустое и **уникальное**; попытка задать пустое или уже занятое
    имя отклоняется с понятным сообщением.
  - Новый профиль создаётся со значениями по умолчанию (§5.1) и сразу становится активным.
  - **Удалить последний профиль нельзя** — как минимум один должен остаться
    (по аналогии с непустым списком клавиш, §3.4). Удаление подтверждается диалогом;
    после удаления активным становится первый профиль в списке.
- **Переключение профиля во время работы кликера останавливает кликер** (нажатые
  клавиши отпускаются): новый профиль может задавать другие клавиши, режим и задержку.
  Для запуска пользователь заново нажимает триггер (уже нового профиля).
- Активный профиль сохраняется в конфиге и восстанавливается после перезапуска (§5).

## 4. UI (Swing)

Одно главное окно, компактная компоновка:

```
┌────────────────────────────────────────────────────────────────┐
│ ClickerCat                                                     │
├────────────────────────────────────────────────────────────────┤
│ Профиль:  [+][−] [ По умолчанию ▼]              [    ✏️    ]    │
│                                                                │
│ Горячая клавиша:      [ F6 ]                    [    ✏️    ]    │
│ Аварийная остановка:  [ ESC ]                   [    ✏️    ]    │
│                                                                │
│ Режим:  (•) Удержание   ( ) Переключение                       │
│                                                                │
│ Задержка между итерациями: [ 100 ] мс (0–10000)                │
│                                                                │
│ [x] Поверх всех окон                                           │
│                                                                │
│ Нажимать клавиши:                                              │
│  ┌────────────────────────────────┐             [    +     ]    │
│  │ Mouse LEFT                     │             [    −     ]    │
│  │ Key SPACE                      │                             │
│  └────────────────────────────────┘                             │
│                                                                │
│ Статус: [ Ожидание ]                            [ Вкл/Выкл ]   │
└────────────────────────────────────────────────────────────────┘
```

Требования к UI:
- Все долгие/блокирующие операции (циклы кликов, перехват) — вне Event Dispatch Thread (EDT).
  Обновление UI — только через `SwingUtilities.invokeLater`.
- Индикация состояния (§3.10): отключён / ожидание / активен.
- Кнопка **«Вкл/Выкл»** — главный выключатель механизма (§3.10); её надпись
  отражает следующее действие («Вкл», когда выключено; «Выкл», когда включено).
- Флажок **«Поверх всех окон»** — держит главное окно поверх остальных; состояние
  сохраняется в конфиге (`alwaysOnTop`, в составе профиля).
- Строка **«Профиль»**: узкие кнопки «+» (создать) и «−» (удалить) слева от выпадающего
  списка — так они не расширяют окно; кнопка «✏️» (переименовать) — в правой колонке
  кнопок. Выбор другого профиля перезаполняет все остальные виджеты окна значениями
  этого профиля.
- Кнопки правой колонки («✏️» назначения и переименования, «+» / «−» списка клавиш,
  «Вкл/Выкл») — **одной ширины**, их края выстроены по вертикали. Кнопки «+» / «−»
  списка клавиш расположены столбиком, сверху вниз.
- Кнопки-символы не растягиваются по ширине ячейки; смысл каждой поясняет всплывающая
  подсказка (tooltip).

### 4.1. Системный трей

- Если ОС поддерживает системный трей, приложение размещает в нём значок с
  контекстным меню: показать/скрыть окно, включить/выключить механизм (§3.10),
  выход. Значок отражает состояние механизма (включён/выключен). Двойной клик
  по значку показывает и поднимает окно.
- **Закрытие главного окна** (крестик) при наличии трея показывает диалог с выбором:
  - **Свернуть в трей** — окно прячется, приложение продолжает работать в трее;
  - **Выйти** — приложение полностью завершается;
  - **Отмена** — окно остаётся открытым, ничего не происходит.
- Если трей ОС недоступен, закрытие окна завершает приложение (диалог не показывается).

## 5. Формат конфигурации (YAML)

Файл `./clickercat.cfg.yaml`:

```yaml
enabled: true               # главный выключатель механизма (§3.10), общий
activeProfile: По умолчанию # имя активного профиля (§3.11)
profiles:                   # непустой список профилей
  - name: По умолчанию      # непустое уникальное имя
    hotkey:
      type: KEYBOARD        # KEYBOARD | MOUSE
      code: VC_F6           # код клавиши JNativeHook (или BUTTONn для мыши)
    emergencyStop:
      type: KEYBOARD
      code: VC_ESCAPE
    mode: HOLD              # HOLD | TOGGLE
    delayMs: 100            # задержка между итерациями, 0..10000
    keys:                   # список нажимаемых клавиш/кнопок (не пустой)
      - type: MOUSE
        code: BUTTON1       # левая кнопка мыши
      - type: KEYBOARD
        code: VC_SPACE
    alwaysOnTop: false      # окно поверх всех остальных
```

Обратная совместимость:
- Поля `alwaysOnTop` и `enabled` необязательны: если их нет в файле, применяются
  значения по умолчанию (§5.1) — окно не поверх всех, механизм включён.
- **Старый формат без профилей** (настройки лежат прямо в корне файла) читается
  как единственный профиль с именем «По умолчанию»; при первом же сохранении файл
  перезаписывается в новом формате.
- `activeProfile` отсутствует или не совпадает ни с одним профилем (ручная правка
  файла) — активным становится первый профиль в списке; это не ошибка.

### 5.1. Значения по умолчанию (поля не могут быть пустыми)

| Поле | Значение по умолчанию |
|------|-----------------------|
| `profiles` | один профиль с именем `По умолчанию` |
| `activeProfile` | `По умолчанию` |
| `enabled` | `true` (механизм включён) |
| `name` | `По умолчанию` |
| `hotkey` | `KEYBOARD / VC_F6` |
| `emergencyStop` | `KEYBOARD / VC_ESCAPE` |
| `mode` | `HOLD` |
| `delayMs` | `100` |
| `keys` | `[ MOUSE / BUTTON1 ]` (один элемент — ЛКМ) |
| `alwaysOnTop` | `false` |

### 5.2. Обработка повреждённого/невалидного конфига

При старте, если основной файл `clickercat.cfg.yaml` не читается или невалиден,
выполняется следующий сценарий (поведение определяется наличием бэкапа):

1. **Бэкап `clickercat.cfg.yaml.bak` существует и валиден:**
   - Пользователю задаётся вопрос: «Файл настроек повреждён. Восстановить из резервной
     копии?».
   - **Да** → настройки берутся из бэкапа, бэкап становится основным конфигом
     (повреждённый файл перезаписывается восстановленными настройками).
   - **Нет** → переход к п.3 (вопрос о сбросе к значениям по умолчанию).

2. **Бэкап отсутствует или сам невалиден** — трактуется как «бэкапа нет» → переход к п.3.

3. **Бэкапа нет (или пользователь отказался от восстановления):**
   - Показывается уведомление: «Файл настроек повреждён. Сбросить к значениям
     по умолчанию и перезаписать файл?».
   - **Да** → применяются значения по умолчанию (§5.1), файл перезаписывается.
   - **Нет** → программа **завершается аварийно** (ненулевой код выхода),
     повреждённый файл **не перезаписывается** (остаётся как есть для ручного разбора).

## 6. Логика работы кликера

1. Пока механизм выключен (§3.10), горячие клавиши игнорируются и кликер не
   запускается. При включённом механизме по триггеру (с учётом режима
   HOLD/TOGGLE, §3.3) запускается рабочий поток.
2. Цикл, пока кликер активен:
   - нажать (`press`) все элементы из `keys`;
   - отпустить (`release`) все элементы;
   - выдержать `delayMs`;
   - перейти к следующей итерации.
3. При остановке (отпускание клавиши в HOLD, повторное нажатие в TOGGLE или аварийная
   остановка) гарантированно отпустить все ранее нажатые клавиши/кнопки, чтобы они
   не «залипли».

### Маппинг кодов
- Коды JNativeHook (`NativeKeyEvent.VC_*`, `NativeMouseEvent.BUTTON*`) → коды `Robot`
  (`java.awt.event.KeyEvent.VK_*`, `InputEvent.BUTTON*_DOWN_MASK`).
- Необходим модуль-конвертер кодов.
- Раскладка клавиатуры специально не обрабатывается: эмулируется нажатие физической
  клавиши, результат зависит от текущей раскладки пользователя.

## 7. Кроссплатформенные особенности

- **Linux**: JNativeHook стабильно работает на **X11**. На **Wayland** глобальный
  перехват ограничен/недоступен — задокументировать как известное ограничение;
  при невозможности инициализации хука показать понятное сообщение.
- **Windows**: работает без доп. разрешений; антивирус/анти-чит может расценивать
  эмуляцию ввода как нежелательную активность — указать в README.

## 8. Нефункциональные требования

- Память: минимальная; одно окно, один рабочий поток кликера.
- Корректное завершение: дерегистрация хука `GlobalScreen.unregisterNativeHook()` при выходе.
- Логирование JNativeHook приглушить (по умолчанию он шумит в консоль) —
  уровень логгера выставить в `WARNING/OFF`.

## 9. Структура проекта (предлагаемая)

```
clickercat/
├── pom.xml
├── clickercat.cfg.yaml        # создаётся при первом запуске
├── clickercat.cfg.yaml.bak    # бэкап, перезаписывается перед каждым сохранением
└── src/main/java/cat/clicker/
    ├── App.java               # точка входа, инициализация
    ├── ui/MainWindow.java     # Swing-окно
    ├── core/Clicker.java      # рабочий поток, цикл кликов (Robot)
    ├── core/HotkeyService.java# регистрация/обработка глобального хука
    ├── config/Config.java     # POJO модели настроек: профили + общие настройки
    ├── config/Profile.java    # POJO именованного профиля (§3.11)
    ├── config/ConfigStore.java# загрузка/сохранение YAML + бэкап
    └── input/KeyMapper.java   # маппинг кодов JNativeHook ↔ Robot
```

## 10. Сборка и запуск

- `mvn clean package` → `target/clickercat.jar` (fat-jar через `maven-shade-plugin`).
- Запуск: `java -jar clickercat.jar` (требуется установленная JRE/JDK 25).
- `maven-compiler-plugin` с `<release>25</release>`.

## 11. Критерии приёмки

1. Приложение собирается `mvn clean package` и запускается на Linux (X11) и Windows.
2. Глобальная горячая клавиша срабатывает, когда окно не в фокусе.
3. Режимы HOLD и TOGGLE работают согласно §3.3.
4. За одну итерацию нажимается весь список клавиш (клавиатура + мышь), затем задержка,
   затем следующая итерация; клики мыши — в текущей позиции курсора без перемещения.
5. Нельзя назначить одну клавишу в несколько ролей (триггер / аварийная остановка / список).
6. Задержка ограничена 0–10000 мс; поля не могут быть пустыми, есть значения по умолчанию.
7. Настройки сохраняются в `./clickercat.cfg.yaml` при каждом изменении; перед сохранением
   создаётся бэкап `./clickercat.cfg.yaml.bak`; настройки восстанавливаются после перезапуска.
8. Аварийная остановка (настраиваемая, по умолчанию `ESC`) немедленно останавливает кликер.
9. Кнопка «Вкл/Выкл» выключает весь механизм: кликер останавливается, горячие клавиши
   перестают срабатывать; состояние сохраняется и восстанавливается после перезапуска
   (старые конфиги без поля `enabled` — как включённые).
10. При остановке/выходе нет «залипших» клавиш; хук дерегистрируется.
11. Профили (§3.11): можно создать несколько профилей, переключение меняет все настройки
    окна и глобальные горячие клавиши; активный профиль восстанавливается после
    перезапуска; последний профиль удалить нельзя; имена уникальны; переключение во время
    работы кликера останавливает его.
12. Конфиг старого формата (без профилей) читается как единственный профиль
    «По умолчанию» с прежними настройками.
