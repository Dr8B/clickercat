# ClickerCat

Кроссплатформенный автокликер с глобальной горячей клавишей. По нажатию заданной
клавиши за одну итерацию эмулируется набор нажатий клавиш клавиатуры и/или кнопок
мыши, затем выдерживается задержка — и так по кругу.

## Возможности

- Глобальная горячая клавиша (срабатывает, даже когда окно не в фокусе).
- Режимы **Удержание** (работает, пока клавиша зажата) и **Переключение** (нажал —
  включил, нажал ещё раз — выключил).
- Набор нажимаемых клавиш/кнопок (клавиатура + мышь); клики мыши — в текущей позиции
  курсора без перемещения.
- Настраиваемая аварийная остановка (по умолчанию `ESC`).
- Защита от конфликтов: одну клавишу нельзя назначить в несколько ролей.
- Флажок «Поверх всех окон» и сворачивание в системный трей.
- При закрытии окна — выбор: свернуть в трей или выйти.
- Настройки автоматически сохраняются в YAML с резервной копией.

## Сборка

```sh
mvn clean package
```

Артефакт — исполняемый fat-jar: `target/clickercat.jar`.

## Запуск

```sh
java -jar target/clickercat.jar
```

Требуется установленная JDK/JRE 25.

Конфигурация создаётся при первом запуске в **рабочей директории** процесса:

- `clickercat.cfg.yaml` — текущие настройки;
- `clickercat.cfg.yaml.bak` — резервная копия (перезаписывается перед каждым сохранением).

## Разработка в WSL

Проект можно собирать и отлаживать из WSL, но у запуска есть важное различие.

**Запуск внутри WSL (WSLg).** Приложение запустится через `java -jar target/clickercat.jar`
и покажет окно, но `Robot`-клики и глобальные хуки останутся внутри виртуального
X-дисплея WSLg и **не влияют на реальный рабочий стол Windows** (системного трея в
WSLg тоже нет). Это годится для отладки логики, но не для реального использования.
Нужна системная либа:

```sh
sudo apt-get install -y libxkbcommon-x11-0
```

Профили для запуска/дебага внутри WSL — «ClickerCat (запуск)» и «ClickerCat (дебаг)»
в `.vscode/launch.json`.

**Запуск на Windows из WSL (рекомендуется для реального использования).** Чтобы клики
и хоткеи работали в самой Windows, приложение запускается нативной Windows-джавой
(`java.exe`) через WSL-interop:

```sh
scripts/run-win.sh          # собрать (если нужно) и запустить на Windows
scripts/run-win.sh --build  # принудительно пересобрать + запустить
scripts/run-win.sh --debug  # запуск с JDWP на :5005 для отладки
```

Скрипт копирует jar на локальный диск Windows — это обязательно, иначе jnativehook
не может распаковать свою `.dll` с UNC-пути `\\wsl.localhost\...`. Требуется JDK,
установленный на Windows, с `java.exe` в системном PATH.

В VSCode те же действия доступны как задачи (Run Task) — см. `.vscode/tasks.json`,
а профиль «ClickerCat (attach к Windows :5005)» цепляет отладчик к процессу на
Windows. Для attach по `localhost` нужен режим mirrored networking в WSL2
(`.wslconfig` → `networkingMode=mirrored`); иначе укажите в профиле IP Windows-хоста.

### Mirrored networking (для attach-отладки)

Файл `.wslconfig` лежит в профиле Windows — `C:\Users\<имя>\.wslconfig` (требуется
Windows 11):

```ini
[wsl2]
networkingMode=mirrored
```

После правки `.wslconfig` нужно перезапустить WSL — параметры `[wsl2]` относятся ко
всей виртуальной машине, поэтому перезапуска одного дистрибутива недостаточно:

1. Закройте VSCode, подключённый к WSL, и все терминалы WSL (иначе сессия
   поднимется обратно).
2. В **PowerShell или CMD на Windows** (не внутри WSL) выполните:

   ```powershell
   wsl --shutdown
   ```

3. Подождите ~8 секунд и снова откройте WSL/VSCode — дистрибутив стартует с новыми
   настройками.

Проверить, что режим применился, можно так — в mirrored-режиме IP внутри WSL
совпадает с адресами Windows-адаптеров (а не подсеть `172.x` от NAT):

```sh
ip addr show eth0 | grep inet
```

## Системный трей

Если система поддерживает трей, приложение сворачивается в значок с меню
(показать/скрыть окно, старт/стоп, выход); значок отражает состояние кликера,
двойной клик по нему поднимает окно. При закрытии окна крестиком появляется
диалог с выбором — **свернуть в трей** или **выйти**. Если трей недоступен
(например, в WSLg), закрытие окна завершает приложение.

## Известные ограничения и предупреждения

### Linux

Глобальный перехват ввода через JNativeHook стабильно работает на **X11**.
На **Wayland** глобальный хук ограничен или недоступен — при невозможности
инициализации приложение покажет предупреждение и продолжит работу с ручным
управлением (кнопка «Старт/Стоп»); горячие клавиши при этом работать не будут.

### Windows

Дополнительных разрешений не требуется. Антивирус или анти-чит может расценивать
эмуляцию ввода как нежелательную активность — это ожидаемо для автокликеров.
Используйте приложение только там, где это разрешено.

## Конфигурация (YAML)

```yaml
hotkey:
  type: KEYBOARD        # KEYBOARD | MOUSE
  code: VC_F6           # код клавиши JNativeHook (или BUTTONx для мыши)
emergencyStop:
  type: KEYBOARD
  code: VC_ESCAPE
mode: HOLD              # HOLD | TOGGLE
delayMs: 100            # задержка между итерациями, 0..10000
keys:
  - type: MOUSE
    code: BUTTON1       # левая кнопка мыши
```

Если основной файл повреждён, при старте предлагается восстановление из резервной
копии или сброс к значениям по умолчанию.
